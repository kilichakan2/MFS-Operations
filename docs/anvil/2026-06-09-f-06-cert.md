## Executive summary

F-06 is the second hexagonal unit in the strangler-fig migration and the first to ship actual
adapter code. It delivers three Supabase adapters (real Postgres), three Fake in-memory adapters
(pure JavaScript Maps, no vendor SDK), and the shared contract test suites that both adapters
pass identically. This is the operational proof of ADR-0002 line 27 and ADR-0003 line 23 in
running code, not just prose.

The architectural significance of this unit is that it establishes the **worked example** that
every future adapter PR copies. F-13 (Users), F-14 (Routes), F-15 (Pricing), F-16 (Cash),
F-17 (Complaints), F-18 (Visits), F-19 (HACCP), F-20 (Admin) — all eight will mirror the
three-directory layout, the contract-suite factory signature, the symmetric Supabase / Fake
wrapper files, and the vendor-type boundary discipline documented in this cert. Getting F-06
wrong copies the error into eight future units. This cert shows it is right.

Two findings are load-bearing for all future audits:

1. **Adapter purity (L8):** `@supabase/supabase-js` appears in exactly three files, all inside
   `lib/adapters/supabase/`. It appears in zero files in `lib/adapters/fake/` or
   `lib/ports/__contracts__/`. ADR-0002 line 27's vendor-type boundary is empirically confirmed,
   not just asserted.

2. **Contract substitutability (L10):** The same 43 contract cases — the exact same describe
   blocks, the exact same test names — pass against the Supabase adapter (real Postgres, live
   local stack) and against the Fake adapter (in-memory Maps, no DB). This is the operational
   meaning of "swap the database vendor = write one new adapter + change one config line."

F-06 ships fully unused. Zero production code paths import any `@/lib/adapters` path.
`lib/services/` and `lib/usecases/` do not yet exist. F-07 wires `OrdersService`;
F-08 rewrites the routes.

## Risk residue from the plan

The plan's §5 lists the following risks. Each is addressed:

**Risk 1 — F-TD-03 blocking cert (HTTP 500s from dev server / ENV injection).** The contract
tests bypass the HTTP layer entirely. L2a shows 43/43 Supabase adapter tests green despite the
three F-TD-03 test files continuing to fail. Root cause isolation confirmed: F-TD-03 is caused
by the `api()` HTTP helper + dev server lifecycle, not by adapter correctness. The calibrated
L2 bar (zero new failures) is met. Discharged for F-06; carried to F-08.

**Risk 2 — TOCTOU race in `findOrderById` re-throw after NotFoundError.** The adapter's
`updateOrder` calls `this.findOrderById` after the patch and re-throws `NotFoundError` if the
read-back returns null (indicating a concurrent delete). This is a race-window advisory: in
production, a concurrent delete between the PATCH and the read-back would surface as a
`NotFoundError` thrown from an operation that actually succeeded. L2a's "updateOrder > throws
NotFoundError when id does not exist" test exercises the synchronous (no race) path. The async
race is not testable deterministically in a contract suite without artificial delays. Carry-forward
advisory: the port JSDoc should document this re-throw as "may throw NotFoundError post-patch if
concurrent delete" — tighten in F-07 or a dedicated JSDoc PR. Not a blocker.

**Risk 3 — `updateOrder` empty no-op still does a read-back.** When `patch` is empty and
`lineReplacement` is undefined, the Supabase adapter skips the PATCH and line-replace calls but
still calls `findOrderById` for the return value. This is a deliberate design (returns the
current state for consistency) but costs one extra round-trip. L2a's "no-op when patch is empty
and lineReplacement is undefined" case passes. By-design behaviour; carry-forward note for
F-07's service layer to avoid redundant read-backs when it knows nothing changed. Not a blocker.

**Risk 4 — `this.findOrderById` JS-this binding fragility (factory-as-object-literal).** If
the adapter is destructured (`const { updateOrder } = createSupabaseOrdersRepository(client)`),
the `this.findOrderById` internal call would lose its `this` context and throw. The plan
accepted this as a known factory-as-object-literal quirk; the default singletons are used as
full objects, not destructured. Advisory for F-07: always call adapter methods on the object
instance, never destructure. Not a blocker.

**Risk 5 — ADR-0004 service-role posture (bypasses RLS).** F-06's Supabase adapters use
`supabaseService` (the service-role client) consistent with today's route behaviour and
ADR-0004. Per-request authenticated clients (F-RLS-03) and the RLS rewrite track (F-RLS-04+)
are independent. L2a contract tests do not set up per-test auth contexts — they exercise the
service-role path, which is the documented current posture. Discharged at the documented risk
level.

**Risk 6 — `ProductsRepository.contract` reorder case is tautological with one seed.** The
"does NOT guarantee result order matches caller-passed id order" case passes vacuously when
the seed contains only one product (no ordering to validate). Advisory for F-13+: when copying
this contract pattern, ensure multi-row seeds for order-sensitive contracts. Not a blocker for
F-06 (single-product seed is correct per the Products port JSDoc, which is silent on ordering).

**Risk 7 — Plan file untracked.** `docs/plans/2026-06-09-f-06-orders-supabase-adapters.md` is
untracked on `main` at ANVIL Lock time (it is in the working tree but not committed to the
branch). Conductor handles this with a docs trail commit at Gate 4 (ANVIL Lock), matching the
F-05 pattern. Not a blocker for the cert.

## Carry-forward advisories (Guard's 7 items)

These are green-flag advisory items only — none are blockers. They are documented here for
F-07/F-13+ planners who will build on F-06's patterns.

1. **Runtime early-return vs `it.skipIf` pattern.** Several Fake adapter contract cases use a
   vacuous early return (`if (!supportsFeature) return`) rather than `it.skipIf`. This works but
   produces a passing test with no assertions, which can mask intent. When tsc supports
   `it.skipIf` for the test-runner version in use, migrate to explicit skips so the intent is
   visible in the test output. Revisit in F-07 or a dedicated test-quality PR.

2. **`findOrderById` re-throwing NotFoundError after race.** The port JSDoc should be tightened
   to document the post-patch re-throw scenario (see Risk 2 above). F-07's `OrdersService` or a
   JSDoc-only PR should add: "May throw `NotFoundError` if a concurrent delete occurs between the
   patch and the read-back." Current JSDoc is silent on this edge.

3. **`updateOrder` empty no-op still does a read-back.** The Supabase adapter always calls
   `findOrderById` as the return value, even when no rows were modified. By-design for
   consistency; F-07's service layer should cache or short-circuit if it detects an empty patch
   to avoid redundant round-trips.

4. **`this.findOrderById` JS-this binding fragility.** See Risk 4 above. F-07 planner: always
