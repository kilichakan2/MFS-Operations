## Executive summary

This PR eliminates the only remaining ad-hoc Supabase SDK client construction outside `lib/supabase.ts`. Previously, `lib/road-times.ts` built its own `createClient(...)` with the same URL and key as the central `supabaseService` — a duplicated client that the v1.2 architecture review flagged as Phase 0 technical debt. The four-line client block is replaced by a direct import of `supabaseService` from `lib/supabase.ts`, using exactly the same credentials and producing observably identical query behaviour. Two `console.warn` and `console.log` calls in the same file are migrated to the structured `log.warn` and `log.info` from `lib/observability/log.ts` (shipped by F-FND-03), emitting JSON-line logs that Vercel ingests cleanly. The intentional swallow-and-fallback contract is preserved verbatim: on DB error, `loadRoadTimes()` returns an empty-but-callable `RoadTimeMatrix` so `exactTSP` falls back to haversine distance rather than propagating the error and crashing the daily delivery-route optimisation. A new integration test covering three cases proves this contract end-to-end against the F-INFRA-01 local Supabase stack.

## Risk residue from the plan

The plan's section 5 lists ten risks. Each is addressed as follows:

**Risk 1 — Swallow-and-fallback contract.** Discharged. Integration test case (c) asserts the empty-matrix shape and the no-throw invariant directly against a live PostgREST error. The `// CONTRACT:` comment added above the error-branch return in `lib/road-times.ts` makes the invariant legible to future readers of the source. No residue.

**Risk 2 — DB-error simulation strategy.** Discharged. Strategy (A) — malformed (non-UUID) IDs in the `.in()` filter — was chosen and works correctly. PostgREST rejects the filter with a UUID-syntax error; the SDK surfaces `{ data: null, error: { message: '...' } }`; `loadRoadTimes()` enters the error branch and returns the empty matrix. Confirmed by test passage. No residue.

**Risk 3 — Schema verification.** Discharged. The seed insert succeeded (3 rows: `A→B`, `B→A`, `A→HUB`), which is conclusive evidence the schema matches the expected shape (`from_id uuid`, `to_id uuid`, `duration_s integer`, `distance_m integer` — all NOT NULL). No residue.

**Risk 4 — Structured logger emits to global console handles.** Discharged. The case (c) spy is scoped inside a `try/finally` and restored before the test exits. The `fileParallelism: false` Vitest config ensures no concurrent test can see the spy state. No cross-test bleed observed. No residue.

**Risk 5 — Optimise route not yet wrapped in `withRequestContext`.** Carried forward as advisory. `loadRoadTimes` log lines will lack `correlationId` until the optimise route is migrated in its own unit. The log lines are structurally correct and strictly better than the previous interpolated strings. This is expected behaviour per F-FND-03's risk documentation. Advisory only; does not affect clearance.

**Risk 6 — Future test isolation for `customer_road_times`.** Carried forward as advisory. Currently no other test writes to this table. The greppable UUID fragment (`f001/f002/f003`) convention is in place. No action in F-01.

**Risk 7 — Slow `supabase start` vs `assertLocalStackReachable` probe.** Not triggered. Stack was running before this ANVIL run began. Already documented in F-INFRA-01. No residue.

**Risk 8 — `error.message` potentially undefined.** Acknowledged, no action. The Supabase SDK consistently returns `PostgrestError` with a `message` string. The case (c) assertion on `parsed.error.length > 0` confirms a non-empty message was present for the UUID-syntax error. No residue.

**Risk 9 — Additional callers of `loadRoadTimes`.** Discharged. Confirmed: one production caller (`app/api/routes/optimise/route.ts:547`); `scripts/test-routing-engine.ts` references `RoadTimeMatrix` by type only and is unaffected. `git diff main app/api/routes/optimise/` returns empty. No residue.

**Risk 10 — `// CONTRACT:` comment adoption.** Discharged. Comment is present in `lib/road-times.ts` lines 49–50 as shipped. Verified by file read. No residue.

The `log.info` / `log.warn` lines emitted by `loadRoadTimes` will appear in Vercel logs without `correlationId` until that route migration happens. Strictly better than before (structured JSON vs interpolated string), but worth noting for the Vercel log-search UX. The owning unit for that migration depends on F-03 (`requireRole` helper). No action in F-01.

The query in `loadRoadTimes` continues to run as service-role (same as before F-01). The open `anon` grants are a separate F-RLS-01 concern, surfaced here so the conductor sees it at Gate 4. No action in F-01.
