## Executive summary

This PR ships the first hexagonal unit in the whole strangler-fig migration. It defines — but
does not implement — the Lego shapes for the Orders bounded context: three port interfaces
(`OrdersRepository`, `CustomersRepository`, `ProductsRepository`) living under `lib/ports/`,
three domain-type modules (`Order` + state-machine, line, filter, patch and create-input
shapes; `Customer`; `Product`) living under `lib/domain/`, and a type-pinning Vitest fixture
at `tests/unit/ports/orders-domain.types.test.ts` that locks every interface field, union
literal, and method signature against accidental contract drift. The PR also adds a Folder
layout + Blockers section to `CLAUDE.md` that aligns the guidance with ADR-0002's naming
convention, while preserving the Lego-principle prose (lines 1-24) byte-for-byte.

No implementations exist yet: no adapters, no services, no use-cases, no route rewrites. The
five Orders routes (`app/api/orders/route.ts`, `app/api/orders/[id]/route.ts`,
`app/api/orders/[id]/picking-list/route.ts`, `app/api/kds/orders/route.ts`,
`app/api/kds/lines/[lineId]/done/route.ts`) stay verbatim until F-08. This is the correct
ADR-0003 strangler-fig sequencing: ports in F-05, adapters in F-06, service in F-07, route
rewrites in F-08.

The structural significance of this unit extends beyond its line count. F-05 is the template
every subsequent domain unit copies from — F-13 Users, F-14 Routes, F-15 Pricing, F-16 Cash,
F-17 Complaints, F-18 Visits, F-19 HACCP, F-20 Admin, F-21 Dashboard. The file header tone,
JSDoc discipline, design-it-twice annotations, depth-rule evidence, and error contract
documented here are the pattern. Nine future domain units will fork from this one. The cert
therefore audits not just whether the code runs, but whether the structural contract it
establishes is sound.

**Case: "OrderState union admits exactly placed/printed/completed"**

```typescript
it("OrderState union admits exactly placed/printed/completed", () => {
  const placed: OrderState = "placed";
  const printed: OrderState = "printed";
  const completed: OrderState = "completed";
  expect([placed, printed, completed]).toEqual([
    "placed",
    "printed",
```

The `@ts-expect-error` directive is the critical signal: TypeScript confirms at compile time
that `"archived"` is not assignable to `OrderState`. If someone adds a fourth literal to the
union, the `@ts-expect-error` ceases to suppress an error and becomes a tsc error itself —
the pin breaks exactly when the contract drifts.

### Observable failure pattern

Two distinct failure signatures are visible:

1. **~490-640ms timeouts returning HTTP 500** on tests that attempt to POST to DB-touching
   endpoints (`/api/orders`, `/api/orders/[id]`, `/api/orders/[id]/picking-list`). The tests
   that exercise pure middleware routing (redirect-to-login, role check returning 307/401) pass
   in 10-500ms with the correct status codes. Tests that require the dev server to actually
   process a request body and talk to the database receive 500 instead of the expected 201/200/400.

2. **HTTP 404** on tests that depend on a prior test having successfully created a fixture
   order (e.g. "allows sales to edit a placed order" tries to operate on an order id that was
   never created because the create test failed). These 404s are cascade failures downstream of
   the 500s above.

3. **Empty audit log result** in "first print emits a printed audit row" (`expected [] to include 'created'`) — the audit query returns empty because the order that should have been created never was.

The passing tests across all three files are the ones that either: (a) test middleware routing
without touching the DB (redirect, role-rejection), or (b) are the KDS auth-pin endpoint tests
(`POST /api/auth/kds-pin`) which have a self-contained route not dependent on the Orders DB
fixture.

### Suspected root-cause hypotheses

1. **Seed lifecycle / stale dev-server state.** The integration suite boots a dev server
   (`webServer` in `vitest.integration.config.ts`) that is shared across all test files. If the
   dev server fails to warm up the Supabase connection pool before the first DB-touching test,
   it returns 500. The middleware-only tests (which do not touch the DB) pass because they run
   first and succeed before the pool timeout.

2. **ENV drift between branch switches.** The `.env.test.local` injects only 3 keys on
   `orders-crud.test.ts` (logged as `◇ injected env (3) from .env.test.local`) but 0 keys on
   `picking-list.test.ts` and `kds.test.ts` (logged as `◇ injected env (0)`). The asymmetry
   suggests that some required environment variables for those test files are not being injected
   on every run, leaving the dev server with missing keys during the routes that need them.

3. **F-04's `@types/eslint` transitive dep (`@types/json-schema`) affecting test runtime.**
   The F-04 cert documented that `@types/json-schema` surfaces 4 latent tsc errors in
   component files. It is possible (though unconfirmed) that the transitive dep also affects
   module resolution in a way that slows or breaks the integration test harness import graph.

4. **Dev server boot race condition.** The `webServer` config may not wait long enough for
   all routes to be compiled before the first test fires. The 500 on DB-touching endpoints
   could be Next.js returning an error before the route module is ready.

### F-08 owns the fix — contract

**F-08 MUST:**

1. Investigate the root cause of the 500s on DB-touching endpoints FIRST, before rewriting
   any routes. The integration suite failures are a symptom; the root cause must be identified
   and fixed at the test harness level before the new port-backed routes can be certified.

2. Fix the root cause in the integration test harness (likely: seed lifecycle, ENV injection,
   or dev-server boot timing).

3. Rewrite all five Orders routes to use F-05's ports (via F-06's adapters and F-07's service),
   then rewrite the three failing integration test suites against the new port-backed routes.

4. The F-08 cert must show all 49 integration tests green (or document any deliberate scope
   changes to individual test cases with explicit justification).

**F-08 must NOT** merge into main while these suites still show 23/49 failures.

## Risk residue from the plan

The plan's §5 lists the following risks. Each is addressed:

**Risk 1 — `listKdsQueue` as an 8th method, not the locked 7.** The pre-grilled KDS queue
was included as a conditional addition at Gate 1 based on the existing route structure.
L1a case "OrdersRepository method names + arity" pins the count at 8 and lists all 8 names
verbatim. The test compiles against the real interface — if the method were absent the
`type Methods = keyof OrdersRepository` assignment would fail tsc. Discharged.

**Risk 2 — `readonly` on interface fields might be stripped.** Every field across all four
`Order`, `OrderLine`, `OrderFilter`, `OrderPatch`, `CreateOrderInput`, `CreateOrderLineInput`
interfaces carries `readonly`. The type-pin fixture constructs `exampleOrder: Order = { ... }`
with a fully populated shape — if a `readonly` were removed, the fixture would still compile,
but any downstream write `exampleOrder.id = "x"` would succeed in tsc where it should not.
This is a residual risk inherent in structural typing. L1a pins the field shapes; the plan
accepted this limitation. Carried forward as advisory.

**Risk 3 — `lib/orders/types.ts` name collision (co-existence of two `OrderState` definitions).** Confirmed via `git diff main...HEAD -- lib/orders/types.ts` returning empty — F-05 does not touch `lib/orders/types.ts`. The two definitions co-exist. F-08 retires the `lib/orders/types.ts` imports route by route. No residue for F-05.

**Risk 4 — CLAUDE.md alignment edit touching lines beyond 26-58 (accidental edit of Lego prose).** L10 deep-look proves lines 1-24 are byte-preserved. The diff shows only additive lines after line 24. Discharged.

**Risk 5 — F-04's ESLint FREEZE rule blocking `@/lib/domain` imports.** The FREEZE rule
blocks `from '@supabase/supabase-js'`, not `from '@/lib/domain'`. The F-05 port files import
only `import type { ... } from "@/lib/domain"` — these are internal app imports, not
`@supabase/supabase-js`. L3 confirms zero lint violations in F-05 files. Discharged.

**Risk 6 — Type-only imports at runtime (erase vs. keep).** All F-05 imports use
`import type`. No runtime values are imported or exported. L5 (Next.js build exit 0, 113
pages) confirms the TypeScript compiler and Next.js bundler handle the pure-type modules
correctly. Discharged.

**Risk 7 — Integration test failures blocking cert.** The pre-existing 23/49 failures are
documented in §F-TD-03 above. The calibrated bar ("no new failures attributable to F-05")
is met. L2 is a calibrated pass. Discharged at the calibrated bar; carried forward to F-08
as a load-bearing obligation.
