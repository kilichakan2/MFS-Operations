## Executive summary

This PR ships the third Phase 0 foundation for MFS-Operations. It introduces `requireRole(req, allowedRoles)` — a single typed helper that replaces the hand-rolled role-check pattern repeated across 104 existing route handlers. Today those handlers each read `req.cookies.get('mfs_role')`, hand-roll an `includes()` check, and return a raw `NextResponse.json({ error: 'Unauthorised' }, { status: 401 })`. This helper does all of that in one call: it reads the four `x-mfs-*` request headers that `middleware.ts` already sets, enforces the same multi-role union logic (primary role plus secondary roles, with the secondary-admin literal filtered out), and either returns a fully-typed `Caller` or throws one of two new typed error classes — `UnauthorizedError` (HTTP 401, code `UNAUTHORIZED`) when no identity is present, or `ForbiddenError` (HTTP 403, code `FORBIDDEN`) when identity is present but no matching role was found. Both errors flow through the existing `withErrors` HOF automatically; no wiring change was needed. As a housekeeping move bundled with this unit, the `KNOWN_ROLES` constant and `isKnownRole` type-predicate are moved from `lib/observability/withRequestContext.ts` into `lib/observability/Caller.ts`, placing the `Role` union and its runtime filter in the same file. Behaviour is byte-for-byte identical. The PR ships the helper unused — zero of the 104 existing role-check sites are migrated; those migrations happen incrementally inside their owning domain PRs.

## Deep-look results

### L1a: ghost-admin filter test (case g) — the load-bearing safety invariant

Test name and duration from the live run output:

```
✓ tests/unit/auth/session.test.ts > requireRole > throws ForbiddenError (403) when primary is office and secondary contains admin and allowedRoles is admin only (secondary-admin filter) 0ms
```

This is case (g). The request carries `x-mfs-user-role: office` and `x-mfs-secondary-roles: admin`, and `allowedRoles` is `['admin']`. The helper filtered `admin` out of the secondary list before building the union, leaving only `['office']`. `office` is not in `['admin']`, so `ForbiddenError` was thrown — not `UnauthorizedError`, not a `Caller`. The ghost-admin safety invariant is proven: a user whose `secondaryRoles` contains `admin` is NOT silently elevated. This mirrors the `.filter(r => r !== 'admin')` in `middleware.ts:128` verbatim.

## Risk residue from the plan

The plan's §5 lists ten risks. Each is addressed as follows:

**Risk 1 — KNOWN_ROLES/isKnownRole move may break the existing withRequestContext suite.** Discharged. L1c ran 9/9 cases unchanged after the move. The widened `isKnownRole(v: string | null | undefined)` signature is a strict superset; the existing call site in `withRequestContext.ts` (where `roleHdr` is `string | null`) still type-checks and the behaviour is identical. No residue.

**Risk 2 — correlationId derivation: requireRole vs withRequestContext.** Discharged. Option (b) was chosen and adopted in the implementation: `requireRole` derives `correlationId` using the same reuse-then-generate logic as `withRequestContext.ts:59-63`. Case (e) in the test asserts `correlationId.length > 0`, which confirms a non-empty string is always returned. Advisory carried forward: if a future PR needs both helpers to share the exact same id for a single request without `withRequestContext` wrapping, a `lib/observability/correlation.ts` extraction would be worthwhile — but F-03's use cases are satisfied. No blocking residue.

**Risk 3 — 401 without WWW-Authenticate header.** Carried forward as advisory. RFC 7235 §3.1 recommends `WWW-Authenticate` on 401 responses. F-03's `UnauthorizedError` does not include it, but this is consistent with the existing behaviour of 80+/104 sites that already return 401 without the header. F-03 standardises the existing posture; it does not regress it. Advisory for a future `withErrors` enhancement. No blocking residue.

**Risk 4 — x-mfs-secondary-roles header absent on public paths.** Discharged. Test case (d) supplies `'x-mfs-secondary-roles': ''` and case (a) omits the header entirely — both paths are exercised and pass. The `?? ''` fallback + `filter(s => s.length > 0)` guard handles both correctly. No residue.

**Risk 5 — Standalone unit tests for UnauthorizedError/ForbiddenError.** Discharged. Gate 2 adopted two standalone test files (`tests/unit/errors/UnauthorizedError.test.ts` and `tests/unit/errors/ForbiddenError.test.ts`), mirroring `NotFoundError.test.ts`'s shape verbatim. Both pass 6/6 cases. Every `AppError` subclass now has its own standalone test file — the convention is fully maintained. No residue.

**Risk 6 — Inline makeRequest() helper vs shared test util.** Carried forward as advisory. The 2-line `makeRequest` helper is duplicated in `tests/unit/auth/session.test.ts` (new) and `tests/unit/observability/withRequestContext.test.ts` (existing). A third copy in an adopter PR would justify extracting `tests/helpers/makeRequest.ts`. F-03 does not do the extraction — APOSD §3 depth-over-shallow-modules; two callers is not enough to justify a shared util. Advisory for the first adopter PR that needs a third copy.

**Risk 7 — AppError constructor signature.** Discharged. Both new error classes pass a plain string message and no `{ cause, context }` options — appropriate since the helper is the origin of the throw, not a wrapper. The `context` option remains available to adopter PRs if they want to log structured details (e.g. `allowedRoles`, `observedRole`). No residue.

**Risk 8 — Naming: UnauthorizedError vs UnauthenticatedError.** Discharged. `UnauthorizedError` is locked by the spec, matches the HTTP status name verbatim, and aligns with `ForbiddenError`. The class doc comment is explicit about semantics. No residue.

**Risk 9 — Role import in withRequestContext.ts after the move.** Discharged. The implementation dropped `type Role` from the `withRequestContext.ts` import (it became unused after `isKnownRole` moved to `Caller.ts`). The calibrated `tsc` grep returned empty, confirming the import is clean. No residue.

**Risk 10 — lib/observability/index.ts barrel not widened.** Discharged. The barrel continues to export only `{ Caller, Role, makeCaller }`. `KNOWN_ROLES` and `isKnownRole` are exported from `Caller.ts` directly. `requireRole` and the modified `withRequestContext.ts` both import from `@/lib/observability/Caller` directly. The barrel is not widened. No residue.

## Carry-forward advisory items

1. **F-04 rule-A scope is clean for KNOWN_ROLES.** With `KNOWN_ROLES` and `isKnownRole` now living in `lib/observability/Caller.ts`, F-04's rule A (forbid `@supabase/supabase-js` SDK imports outside `lib/supabase.ts`) is unaffected — these are pure TypeScript exports with no SDK dependency. F-04 lands cleanly.

2. **`deriveCorrelationId` duplication.** `requireRole` and `withRequestContext` both inline the same 6-line logic (reuse `x-request-id` if 1..128 chars, else generate fresh hex). A `lib/observability/correlation.ts` extraction is worth doing when a third caller appears. The conductor should note this at Gate 4 as a Phase 0b debt item for F-TD-01.

3. **The 104 existing role-check sites return 401 for "wrong role" — semantically wrong.** They should return 403 (Forbidden), not 401 (Unauthorized). `requireRole` fixes this correctly at the helper level — adopter PRs that migrate to `requireRole` will silently correct the 104 sites. No action in F-03, but the conductor should call this out at Gate 4 as a key adopter-PR side-benefit: once a route migrates to `requireRole`, its "wrong role" branch automatically upgrades from 401 to 403.

4. **`WWW-Authenticate` header on 401 responses.** F-03 standardises the existing behaviour (no header) across 104 sites. If a future PR adds a real auth challenge flow (e.g. OAuth2 bearer), `UnauthorizedError.toJSON()` or `withErrors.ts` will need to inject the header on 401 responses. Flagged here for the conductor's Phase 1+ planning.

5. **Supabase CLI update available.** Running 2.105.0 (upgraded from 2.102.0 at F-01 cert time). No newer version seen at time of this run. No action.
