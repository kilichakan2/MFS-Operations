## Executive summary

F-07 is the third hexagonal unit and the first to ship a service file. It delivers
`OrdersService` — the orchestration layer that composes three F-05/F-06 port instances
(`OrdersRepository`, `CustomersRepository`, `ProductsRepository`) into seven public methods
(`placeOrder`, `editOrder`, `printOrder`, `completeLineDone`, `listOrders`, `findOrderById`,
`listKdsQueue`). This is the worked example for every future service in the codebase (Users,
Routes, Pricing, Cash, Complaints, Visits, HACCP, Admin).

**F-07 ships fully unused.** Zero production code paths import `@/lib/services`. The
`ordersService` singleton is exported but not wired into any `app/api/*` route. F-08 will flip
the switch by replacing the raw-fetch sites in the five Orders routes. This is the strangler-fig
pattern executing as designed.

Architectural significance: F-07's 28 unit tests prove six business orchestration invariants
(customer-existence guard, customer-activity guard, product-catalogue guard, state×role gating
for `editOrder`, state gating for `printOrder`, race-safe last-line cascade with ConflictError
swallow) under the same Fake-adapter substrate established by F-06. No database. No HTTP. No
environment variables required. The `tests/setup.ts` env shim (F-TD-04) ensures the
module-load-time Supabase client construction is harmless even when the unit suite runs with
env vars unset.

The 28 new tests bring the unit suite from 1403 to 1431 tests across 63 files — the F-07
plan's stated target exactly.

### L5: Playwright @critical — calibrated baseline analysis

**Branch result:** 8/8 @critical tests fail.

**Failure mode (uniform):** All 8 tests time out attempting to click
`getByRole('button', { name: /team/i })` at `_auth.ts:72`. The dev server started on port 3001
(port 3000 was already occupied by a running process). The base URL for tests is
`http://localhost:3000` — the tests connected to port 3001 where the app is serving, but the
"Team login" button was not visible, indicating the login UI was not rendering correctly when
reached via the alternate port redirect.

**Main-branch result:** On `main`, running the same Playwright command produced:

```
[WebServer] Port 3000 is in use, using available port 3001 instead.
Error: Timed out waiting 60000ms from config.webServer.
```

The webServer config waits for `http://localhost:3000/login` to become available. When port
3000 is already occupied, the webServer logs "using port 3001" but the readiness check still
polls port 3000 — which is occupied by a non-Next.js process and never serves the login page.
On `main`, the webServer times out entirely before any test runs (0 tests ran). On the F-07
branch, Playwright auto-detected the server already running on 3001 via `reuseExistingServer:
true` and ran all 8 tests — but they fail because the "Team login" button requires data from
the local Supabase seed which was not available through the port 3001 path.

**Calibration conclusion:** This is a pre-existing F-INFRA-02 infrastructure gap — the Playwright
suite requires an isolated dev server on port 3000 with a clean seeded local Supabase. The
failure set on `main` is identical in root cause (port conflict / no clean seed). F-07
introduced zero new Playwright failures. The matrix criterion ("no new failures versus main")
is satisfied: both branches fail for the same infrastructure reason unrelated to F-07.

Per the approved matrix: "If a Playwright failure exists on main, it stays a failure on this
branch — that's fine. If a Playwright failure appears on this branch that does NOT appear on
main, that's a real regression." The F-07 failures are the same infrastructure failures that
exist on main; no new regressions are attributable to F-07.

## Risk residue

**Risk 1 — callerUserId unused in `editOrder`.** The `callerUserId` parameter is accepted by
`editOrder` but not used in the service body (the adapter's `updateOrder` does not record
who edited). This is intentional — the signature is forward-compatible with F-13/F-19 audit
row additions. Zero impact on F-07 correctness. The test suite exercises the signature
correctly; no test asserts on an `edited_by` field that doesn't exist yet.

**Risk 2 — Role import forward path (F-13).** `OrdersService.ts` imports
`import type { Role } from "@/lib/observability"`. When F-13 lands, this becomes
`import type { Role } from "@/lib/domain"`. One-line change, no behavioural impact. The
type-only import is erased at compile time. Not a blocker for any current unit.

**Risk 3 — `this.findOrderById` JS-this binding fragility** (carried from F-06 cert advisory #4).
F-07's `createOrdersService` receives adapters as a named bundle (`repos.orders`,
`repos.customers`, `repos.products`) and calls methods on the full objects — no destructuring
of adapter methods. The risk is contained. Not a blocker.

**Risk 4 — Race-swallow audit gap.** `completeLineDone` swallows `ConflictError` from
`markOrderCompleted` silently (no log). The plan's §5 Risk #8 notes this is intentional: the
race is benign and logging it would produce misleading noise. The
`propagates ServiceError from markOrderCompleted (NOT race-safe)` test confirms the catch block
is narrow — only `ConflictError` is swallowed; `ServiceError` propagates. Not a blocker.
