# RLS Audit & Threat Model — 2026-06-12

**Unit:** F-RLS-01 (docs only). **Author:** sprint Day 1.
**Source data:** Supabase MCP `get_advisors(security)` + `list_tables` against the
production project `uqgecljspgtevoylwkep`, run 2026-06-12; plus a full
code sweep of client instantiation, route auth, and migrations.
**Governing decision:** ADR-0004 (RLS vs service-role security model).
**Companion:** F-RLS-02 expand-contract plan (`docs/reference/security/rls-expand-contract-plan-2026-06-12.md`).

This document records, per table, the current RLS state, who the legitimate
callers are, and what a missed application-level check would leak. It is the
per-table evidence ADR-0004 calls "the highest-priority parallel work" — until
it exists, the team is reasoning about the accepted risk window without data.

---

## 1. Executive summary

The app runs the model ADR-0004 names: **every API route uses the Supabase
service-role key**, which bypasses Postgres RLS entirely. The database trusts
whatever the application says about the caller. Authn/authz is enforced only at
the HTTP edge (Next.js middleware + per-route role checks). There is no
database-level safety net.

Three findings rank above the rest. The first is new this audit and is the most
serious; the second and third are the advisor's ERROR-level lints.

### Finding 1 (CRITICAL, new) — the session cookie is unsigned plaintext JSON

`app/api/auth/login/route.ts:207` sets `mfs_session` to a plain
`JSON.stringify({ userId, name, role, secondaryRoles })`. `middleware.ts:118`
trusts it with a bare `JSON.parse` — no signature, no MAC, no encryption. The
cookie is `httpOnly` (so XSS can't read it) but nothing stops a logged-in user
from opening devtools, editing their own `mfs_session` cookie to
`{"role":"admin", ...}`, and reloading. Because the entire app is
service-role-first and trusts the middleware-derived role, that forged cookie
grants **full admin** — every route, every table, read and write.

This is a privilege-escalation hole that exists _independently_ of the RLS
state: turning RLS on does not close it, because the app's own queries run as
service-role regardless. It must be fixed as part of the auth track (signed or
encrypted sessions — e.g. HMAC with a server secret, or an encrypted JWT). It
is recorded here because the threat model is incomplete without it, and because
it raises the priority of F-RLS-03 (per-request authenticated client): once the
DB enforces RLS against a _verified_ identity, a forged middleware header stops
being sufficient to reach another user's rows.

### Finding 2 (CRITICAL, advisor ERROR) — 42 tables have RLS disabled and are exposed via PostgREST

The advisor reports `rls_disabled_in_public` at ERROR level for **42 tables**,
with the explicit warning: _"These tables are fully exposed to the anon and
authenticated roles used by Supabase client libraries — anyone with the anon key
can read or modify every row."_

The code sweep found the app does **not** ship `NEXT_PUBLIC_SUPABASE_ANON_KEY`
to the browser today, which is a mitigating factor against casual exposure. But
the anon key is a **public-by-design credential** in Supabase's model — it is not
a secret, and the security boundary is _supposed_ to be RLS. With RLS off, the
PostgREST endpoint (`/rest/v1/<table>`) on the production project will serve
read/write to anyone who presents the anon key, regardless of what this app's
frontend does. The exposure is real; "we don't use the anon key" is a property
of the current client code, not a guarantee of the database.

The exposed set includes high-sensitivity data (see §3): financial records
(`cash_entries` 157 rows, `cheque_records`), commercial pricing
(`price_agreement_lines` 277 rows), and **staff health records**
(`haccp_health_records` — GDPR special-category data).

### Finding 3 (WARN, advisor) — SECURITY DEFINER functions callable by anon

Four `SECURITY DEFINER` functions are executable by the `anon` and
`authenticated` roles via `/rest/v1/rpc/<fn>`: `is_admin()`,
`orders_audit_trigger()`, `order_lines_audit_trigger()`, and — most concerning —
`replace_agreement_lines(p_agreement_id uuid, p_lines jsonb)`, which **mutates
pricing data** and runs with the definer's elevated rights. The two audit
triggers are not meant to be RPC-callable at all. `is_admin()` leaks whether an
arbitrary `app.current_user_id` is an admin. These need `EXECUTE` revoked from
`anon`/`authenticated`, or to be switched to `SECURITY INVOKER`.

### Secondary advisor findings

- `rls_enabled_no_policy` on `order_idempotency_keys` — RLS on, zero policies =
  deny-all to anon (service-role still bypasses). This is **safe** (correct for a
  service-only ledger); the lint is informational. No action beyond a comment.
- `function_search_path_mutable` (WARN) on `set_updated_at`,
  `replace_agreement_lines`, `haccp_search`, `generate_order_reference` — set a
  fixed `search_path` to prevent search-path injection. Fold into F-RLS-04 slices.
- `extension_in_public` (WARN) — `pg_net` installed in `public`. Low priority;
  move to a dedicated schema during the RLS track.

## 2. Current security model (the baseline ADR-0004 migrates from)

- **One client.** `lib/supabase.ts` constructs a single service-role
  `createClient` singleton (server-only). The code sweep found **no** browser-side
  Supabase client and **no** remaining inline-key `fetch` paths reaching the DB
  (the 14 inlined paths from Critical Finding C2 appear cleaned up; the raw-fetch
  sites that remain are tracked per-domain in the roadmap and go through routes).
- **Auth.** Custom session cookie, not Supabase Auth JWTs. Login verifies a
  bcrypt credential against `users`, then sets `mfs_session` (see Finding 1).
  `middleware.ts` parses the cookie and injects `x-mfs-user-*` headers; routes
  call `requireRole()` (`lib/auth/session.ts`) against those headers.
- **Roles.** `user_role` enum: `warehouse, office, sales, admin, driver, butcher`.
  Path→role allow-list lives in `middleware.ts`. KDS is a public kiosk (butchers
  PIN-authenticate per action via `/api/auth/kds-pin`).
- **`app.current_user_id` is never set by the app.** The order-pipeline RLS
  policies and audit triggers read `current_setting('app.current_user_id')`, but
  no route issues `SET LOCAL`, so audit rows are written with `user_id = NULL`
  (attribution gap, deferred to the order-pipeline RLS slice / F-RLS-03).

## 4. Threat model

| #   | Threat                                        | Vector                                                           | Today's control                   | Gap                                                                                 |
| --- | --------------------------------------------- | ---------------------------------------------------------------- | --------------------------------- | ----------------------------------------------------------------------------------- |
| T1  | Privilege escalation to admin                 | Edit own `mfs_session` cookie, set `role:admin`                  | none (unsigned cookie)            | **OPEN — Finding 1; fix in auth track, sign/encrypt the session**                   |
| T2  | Direct data exfiltration / tampering          | anon key + PostgREST `/rest/v1/<table>` on the 42 RLS-off tables | none at DB                        | **OPEN — Finding 2; F-RLS-04 slices enable RLS+policies**                           |
| T3  | Pricing tampering via RPC                     | anon calls `replace_agreement_lines`                             | none                              | **OPEN — Finding 3; revoke EXECUTE / SECURITY INVOKER**                             |
| T4  | Cross-user data leak via a missed route check | one forgotten `requireRole()` among ~88 routes                   | hand-rolled per-route checks only | partial — no DB net until F-RLS-final; mitigated by `requireRole()` standardisation |
| T5  | Search-path injection in SECURITY DEFINER fns | mutable `search_path`                                            | none                              | low — fix during F-RLS-04 slices                                                    |
| T6  | Audit trail can't attribute actions           | app never sets `app.current_user_id`                             | none                              | `order_audit_log.user_id` NULL; fix with F-RLS-03 authenticated client              |

**Priority order for remediation:** T1 (auth track, independent of RLS) and T2
(RLS slices) are co-critical. T3 is a fast standalone fix. T4/T6 close as the
authenticated client (F-RLS-03) and slices land. T5 rides each slice.

## 5. Accepted risk window (per ADR-0004)

Between now and F-RLS-final, the production DB remains "trust the application"
for the app's own queries, **and** the 42 RLS-off tables remain directly
exposed via PostgREST. ADR-0004's named mitigations hold (standardised
`requireRole()`, the F-04 service-role lint freeze, this audit's visibility),
but they do **not** cover T1, T2, or T3 — those are net-new findings from this
audit that the ADR's risk-window text predates.

**Recommendation to surface at the next gate:** T1 (unsigned cookie) and T3
(anon-callable pricing mutation RPC) are cheap, high-impact fixes that do not
require the full RLS machinery. Consider pulling them forward rather than
waiting for their domain slice. T2 is the RLS track proper and follows the
expand-contract plan in F-RLS-02.
