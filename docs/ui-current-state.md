🗣 In plain English: the app is a modern React/Next site styled almost entirely with Tailwind

There is exactly one global stylesheet. Icons come from one library for navigation but are
otherwise drawn by hand as raw SVG in nearly every screen — there is no shared icon set.

### 1.1 · The token foundation exists in TWO places (known duplication to fix)

The design tokens are declared **twice**, with identical values, in two different mechanisms:

1. **`tailwind.config.ts`** — as Tailwind `theme.extend` entries (`colors`, `fontFamily`,
   `fontSize`, `maxWidth`, `borderRadius`, `boxShadow`, `transitionDuration`,
   `transitionTimingFunction`). These power utility class names like `bg-mfs-navy`, `rounded-mfs-lg`.
2. **`app/globals.css` `:root`** — as raw CSS custom properties (`--mfs-navy`, `--mfs-radius-lg`,
   `--mfs-space-4`, etc.). The file's own comment at line 49 admits *"duplicates the existing
   :root above; same values."*

🗣 In plain English: the same palette/sizes are written out in two separate files that must be
kept in sync by hand. Change a colour in one and forget the other and they silently disagree.
This double-entry is flagged here as a cleanup target for the rebuild — one source of truth is wanted.

Additionally the type ramp is **split across both files**: `tailwind.config.ts` maps each `fontSize`
name to a `var(--text-*-size)`, and `app/globals.css` defines those `--text-*-size` variables
(plus a `@media (min-width: 768px)` block that resizes them on desktop). So the font sizes are
genuinely shared via CSS variables, but the colour/spacing/radius/shadow/motion scales are
hard-duplicated.

🗣 In plain English: text sizes are wired together properly (one place feeds the other and they
even grow on big screens). The rest of the scales are copy-pasted twice.

### 1.3 · NO semantic token layer

Tokens are named by **colour/role at the palette level** (`mfs-navy`, `mfs-orange`, `mfs-success`),
**never by intent** (there is no `action.primary`, `surface.raised`, `text.muted`, `border.default`).
Components reach for raw palette names directly, so the *meaning* of a colour lives only in the
developer's head.

🗣 In plain English: nothing says "this is THE primary button colour" — code just says "navy."
If the brand later decides primary should be a different hue, every `mfs-navy` has to be hunted
down and judged one-by-one ("is this navy *because* it's primary, or just because it's navy?").
A semantic layer is the named-by-purpose layer this app is missing.

### 3.3 · What does NOT exist

- **No `components/ui/`** directory (confirmed — glob returns nothing).
- **No shared `Button`, `Input`, `Select`, `Card` (outside the dashboard primitives), `Modal`,
  `Badge`, `Table`, `Toast`** primitives. Buttons/inputs/pills are re-hand-rolled with raw Tailwind
  on every screen.
- **No headless component library** (no Radix, no React Aria).
- **No shadcn/ui**, no MUI, no Chakra, no component framework of any kind.

🗣 In plain English: outside that one dashboard kit, there are no shared building blocks. Every
screen re-draws its own buttons, inputs, badges and modals from scratch in Tailwind. That is the
single biggest reason the look drifts between screens — and the main thing the rebuild fixes.

## 5 · Style sprawl (the rebuild target)

The token system exists but adoption is partial. Three problems coexist: raw hex literals, inline
`style={{}}` blocks, and heavy use of Tailwind's **default** palette (`bg-red-400`, `bg-amber-400`,
`slate-*`) instead of `mfs-*` tokens.

🗣 In plain English: most screens were built before/around the token system and never converted.
They hard-code colours as hex strings, sprinkle one-off inline styles, and use Tailwind's stock
colours — so the same "red" or "amber" is spelled five different ways across the app.

### 5.2 · Inline `style={{}}` blocks — **102 across 32 files** (scan-time count)
Mostly legitimate dynamic values (progress-bar `width`, `gridTemplateColumns`, safe-area insets,
Leaflet sizing, `translateZ` compositing hacks). Worst: `components/AppHeader.tsx` (safe-area padding),
`components/BottomNav.tsx` (3 — touch/compositing), `app/haccp/*` (faux-viewport popups, progress bars),
leaflet adapters. Note the HACCP popups deliberately use inline `position:absolute` "faux viewport"
patterns (`app/haccp/page.tsx:709`).
🗣 In plain English: inline styles are the per-element style overrides bolted straight onto the JSX.
Most here are unavoidable (a progress bar's width has to be computed), but they're another place the
rebuild must account for rather than assume everything is a class.

### 5.3 · Raw default Tailwind palette — **1070 `bg-{color}-{n}` across 50 files** (scan-time count)
Screens lean on Tailwind's stock palette instead of tokens. Worst: `app/haccp/audit` (153),
`app/haccp/delivery` (86), `app/haccp/mince` (67), `app/haccp/annual-review` (62), `app/haccp/reviews`
(52), `app/admin/page` (46), `app/haccp/page` (42), `app/haccp/process-room` (40),
`app/haccp/product-return` (40), `app/haccp/people` (38)… Even the chrome does it: **`AppHeader`'s
`SyncDot` uses `bg-red-400` / `bg-amber-400`** (`components/AppHeader.tsx:36,42`) and
`RecentActivity`'s sync dot uses `bg-green-400` / `bg-amber-400` — the offline indicator's colours are
NOT tokens. Status pills everywhere use `bg-amber-100 text-amber-700`, `bg-green-100`, `bg-red-100`, etc.
🗣 In plain English: status colours (open=amber, resolved=green, error=red) are spelled with
Tailwind's built-in colours, not the brand's. Even the little "syncing" dot in the header is a stock
amber, not a token. Semantic status colours are a big chunk of what a semantic token layer would capture.

### 5.4 · The good news — partial token adoption (~194 `mfs-*` class usages across 17 files)
The chrome (`AppHeader`, `BottomNav`, `RoleNav`, `DesktopSidebar`, `MoreDrawer`) and the entire
**admin dashboard** (`primitives.tsx` 27, `cards.tsx` 42, `stat-blocks.tsx` 9, `page.tsx` 3, plus the
`/admin/*` drill-downs) DO use `mfs-*` tokens correctly. So the migration target is: bring the ~50
hex/raw-palette files up to the standard the dashboard already meets.
🗣 In plain English: the navigation frame and the admin dashboard already "speak token." They prove
the system works — the job is to convert the rest of the app to match them.

## 6 · Cross-cutting UI concerns the rebuild MUST preserve

🗣 In plain English: these are app-wide behaviours that aren't tied to one screen. They're easy to
break in a redesign because they live in the seams — so each is called out explicitly to protect it.

**PWA / offline.**
- `components/PwaGuard.tsx` (mounted in `app/layout.tsx`) redirects a backgrounded standalone app
  back to a role-appropriate URL.
- **SyncDot** lives in `AppHeader` and reads `useUnsyncedQueue()` from `lib/wiring/localCache` —
  shows amber pulse while syncing, red ring when ≥3 retries (stuck), nothing when clear.
- `components/RecentActivity.tsx` reads the offline Dexie queue reactively (`useLiveQuery`) and shows
  per-row synced/unsynced dots; works with no network.
- The manifest, theme-color `#16205B`, apple-mobile-web-app meta and icons are wired in `app/layout.tsx`.
🗣 In plain English: this app is installable and works without signal — the header dot and the
"my activity today" list are how staff know whether their taps have reached the server yet. A redesign
that drops the sync dot or the offline list would silently hide data-loss risk from warehouse staff.

**Android / Capacitor packaging.** `@capacitor/core|android|cli ^8.3.3` (`package.json:27–29`) — the app
ships as an **Android APK**. Label printing is Android-native via the **Sunmi V3** device; the UI hook
is `components/PrintLabelStrip.tsx` (orange 100mm AirPrint/iframe + blue 58mm Sunmi silent path).
🗣 In plain English: besides running in a browser, the app is also wrapped as an Android app for the
warehouse's Sunmi label-printer handhelds. The two print buttons map to two real printers. (Per project
memory, label printing is a known-fragile area scheduled for attention after the current sprint.)

**Login keypad.** `components/AuthKeypad.tsx` — a bespoke 4-digit PIN pad (dots, haptics, auto-submit,
physical-keyboard fallback, error pulse). Used by both `/login` and the `/haccp` kiosk door.
🗣 In plain English: people sign in by tapping a 4-number PIN, not typing a password. It's a distinct
input pattern (think phone-unlock screen) that the design system must keep as a first-class component.

**Leaflet maps.** `lib/adapters/leaflet/MapCanvas.tsx` + `MarkerMapCanvas.tsx` are the ONLY files that
import `leaflet`/`react-leaflet`; they inject their own DOM/CSS (`leaflet/dist/leaflet.css`, CDN marker
icons, hand-built `divIcon` SVG pins, OpenStreetMap tiles). `MapView`/`RouteMap`/`MapTabContent` are
vendor-neutral shells over them. Maps resist tokenisation (popups use inline styles, pin colours are
passed as raw hex like `#EB6619`).
🗣 In plain English: the maps are drawn by an outside library that brings its own styling and paints
straight onto the page. The app deliberately quarantines it behind two adapter files. The redesign can
restyle the surrounding page but largely can't tokenise inside the map itself. There is a known,
deferred "Map container already initialized" re-mount bug to be aware of.

**Print surfaces.** `PrintLabelStrip` (label printing), the orders **picking-list** print path, and
PDF/spreadsheet exports via `jspdf` + `jspdf-autotable` (`package.json:34–35`) and `xlsx` (line 48,
cash export). These are separate print/export render paths, not normal screens.
🗣 In plain English: some "screens" are actually things that get printed or downloaded (labels,
picking lists, PDF/Excel exports). They have their own layout rules and must not be styled as if they
were on-screen web pages.

**i18n / Turkish.** `lib/LanguageContext.tsx` + `useLanguage()` provide EN/TR with `t(key)`, persisted
in `localStorage` (`mfs_lang`); toggle lives in `AppHeader` (mobile DotMenu flag + desktop EN|TR pill).
Server renders English first to avoid hydration mismatch. **Known gap:** many strings are hardcoded
literals (e.g. driver's "My Route"/"Kudos", and most page bodies) — TR coverage is partial and a
per-section translation audit is planned.
🗣 In plain English: the app supports English and Turkish with a toggle in the header, but lots of
text was never run through the translator, so Turkish is incomplete. The rebuild should route all new
text through `t()` and expect a translation cleanup pass.

## 7 · Conventions (for the implementer's build)

- **File structure.** Presentation in `app/**` (App Router pages) + `components/**`; per-route private
  parts under `app/<route>/_components/`. Hexagonal core in `lib/domain|ports|services|usecases|adapters|wiring`
  (see project `CLAUDE.md`). UI imports services/use-cases via `lib/wiring/*`, never adapters directly.
- **Client components.** Almost every page/component is `'use client'` (top of file). State is local
  React (`useState`/`useEffect`/`useCallback`); data via `fetch('/api/...')`; offline via Dexie hooks
  in `lib/wiring/localCache`.
- **Styling.** Tailwind utility classes assembled with `[ ... ].join(' ')` arrays for conditional
  variants (no `clsx`/`cva`). Variants typed as string-literal unions on props (e.g. `Accent`,
  `ModalType`, `TileState`, `maxWidth`).
- **Responsive.** Single `md` breakpoint switches chrome (`md:hidden` ↔ `hidden md:block`); `<body
  data-mfs-chrome="true">` triggers the desktop sidebar padding rule in `globals.css:162–166`.
- **Roles.** Source of truth = `middleware.ts` (`ROLE_PERMISSIONS`, `ROLE_HOME`, `PUBLIC_PATHS`),
  mirrored by `RoleNav.buildMatrix`/`buildSidebarItems` and `PwaGuard`. Role read client-side from the
  `mfs_role` cookie.
- **Modals/sheets.** Hand-rolled (no portal lib): fixed overlay + ESC handler + backdrop-click close +
  CSS transform transitions (`DetailModal`, `BottomSheetSelector`, `MoreDrawer`).
- **Icons.** lucide-react for nav; inline hand-coded `<svg>` (often `currentColor`) everywhere else.

🗣 In plain English: pages are mostly self-contained client React files that fetch their own data and
build their look from Tailwind class strings, with conditional styling done by joining arrays. Who-sees-what
is decided in one middleware file and mirrored into the menus. Popups are all built by hand. Knowing these
patterns lets the rebuild edit in the same grain instead of fighting it.

## 8 · Gaps & inconsistencies a redesign should clean up

1. **Token double-declaration** — same palette/scale values live in both `tailwind.config.ts` and
   `globals.css` and must be hand-synced (the file even admits it). Pick one source of truth.
2. **No semantic layer** — colours named by hue, not intent; status colours (`amber`/`green`/`red`)
   are raw Tailwind defaults, not tokens — even the header SyncDot.
3. **No shared primitive library** — `Button`/`Input`/`Badge`/`Modal`/`Table` re-hand-rolled per
   screen; the only kit (`dashboard/admin/_components/primitives.tsx`) is route-private and should be
   promoted to a shared library.
4. **653→664 hex literals + 1070 raw-palette classes** across ~50 files — the bulk of the migration.
5. **Font mismatch** — `GTF Adieu` declared but not shipped (falls back to Inter); headings actually
   render in Plus Jakarta Sans (wired in raw CSS, not a token); token name ≠ rendered font.
6. **Owned `--mfs-space-*` scale unused** — screens use Tailwind default spacing instead.
7. **Two partially-tokenised dark contexts** — KDS (`mfs-kds-*`) and HACCP kiosk (`slate-*`) overlap
   conceptually but are implemented differently; unify under one theming approach.
8. **Legacy route names** — `/screen2`, `/screen3` coexist with `/complaints`, `/visits`; clarify.
9. **Partial Turkish coverage** — many hardcoded strings bypass `t()`.
10. **Map + print surfaces** resist tokenisation (inline styles, vendor CSS) — plan for them explicitly.

🗣 In plain English: the cleanup list, in priority order — collapse the duplicated tokens into one
place, add a "named-by-purpose" colour layer (so status reds/ambers/greens are tokens), build the
shared button/input/badge kit the app never had, then convert the ~50 screens off hand-typed colours,
and finally tidy fonts, dark themes, old route names, and Turkish gaps.
